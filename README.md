# Семантичний та гібридний пошук наукових статей arXiv

## Опис проєкту

Цей проєкт реалізує пошук наукових статей за метаданими arXiv із використанням векторних представлень тексту та гібридного ранжування результатів.

У межах роботи виконується повний пайплайн:

1. Підготовка підмножини даних із початкового набору arXiv.
2. Побудова dense embeddings для назв і анотацій наукових статей.
3. Завантаження векторів і метаданих до Pinecone.
4. Реалізація семантичного пошуку та пошуку з metadata-фільтрами.
5. Дослідження впливу різних стратегій chunking.
6. Реалізація лексичного пошуку за допомогою BM25.
7. Об'єднання результатів семантичного та лексичного пошуку через Reciprocal Rank Fusion (RRF).

## Мета роботи

Мета роботи — на практиці дослідити принципи побудови системи пошуку за текстовими даними та порівняти:

* семантичний пошук на основі dense embeddings;
* лексичний пошук на основі BM25;
* гібридний пошук, який поєднує обидва підходи;
* пошук за документами та пошук за їх окремими чанками;
* різні метрики подібності для векторного пошуку.

## Результати та аналіз

### 1. Підготовка даних

#### 1.1. Завантаження і підготовка датасету

Для роботи використано датасет метаданих наукових публікацій arXiv з Kaggle. Початковий файл `arxiv-metadata-oai-snapshot.json` має формат JSON Lines (`JSONL`), де кожний рядок містить окремий JSON-об'єкт із метаданими однієї статті. Оскільки повний файл має значний розмір, для подальших експериментів було сформовано підмножину з `10 000` валідних записів.

Скрипт `scripts/01_prepare_data.py` читає початковий файл послідовно, не завантажуючи весь датасет у пам'ять. Для кожної статті перевіряється наявність назви (`title`) та анотації (`abstract`), після чого зберігаються поля, необхідні для подальшого пошуку:

* `id` — ідентифікатор статті в arXiv;
* `title` — назва статті;
* `abstract` — текст анотації;
* `authors` — відформатований список авторів;
* `year` — рік першої публікації статті;
* `category` — основна категорія статті.


#### 1.2. Вибір інструментів

##### 1. Порівняння Pinecone, Qdrant і Chroma

Для виконання цього завдання використовується **Pinecone**. Це повністю керований хмарний сервіс для векторного пошуку: не потрібно самостійно розгортати сервер, налаштовувати сховище, індексування та підтримку інфраструктури. Для навчального проєкту це є суттєвою перевагою, оскільки дозволяє зосередитися саме на побудові embeddings, формуванні metadata, виконанні запитів і порівнянні результатів. У моєму випадку для початку роботи достатньо було зареєструвати обліковий запис, отримати API-ключ і зберегти його у локальному файлі `.env`. Pinecone є комерційним managed-сервісом; для цього завдання використовується безкоштовний план Starter із відповідними обмеженнями. У стандартному serverless-сценарії дані зберігаються у хмарній інфраструктурі Pinecone, тому такий варіант не підходить для вимоги повністю локального зберігання даних. Водночас для enterprise-сценаріїв Pinecone також пропонує BYOC-підхід, за якого data plane може працювати у хмарному середовищі замовника.

**Qdrant** відрізняється тим, що його основний рушій є open-source рішенням під ліцензією Apache 2.0 і може бути розгорнутий локально або у власній інфраструктурі. При цьому Qdrant не обмежується лише self-hosted режимом: існує також Qdrant Cloud. З практичної точки зору початкове локальне розгортання Qdrant не є складним: його можна швидко запустити через Docker та працювати з ним через REST/gRPC API і Web UI. Я вже мав базовий досвід роботи з Qdrant, і для невеликого або середнього проєкту з вимогами до локального контролю даних обрав би саме його: він має зручний інтерфейс, розвинуту фільтрацію за payload-полями та вбудовану підтримку hybrid retrieval. Водночас у production-середовищі self-hosted варіант потребує самостійного вирішення питань резервного копіювання, безпеки, моніторингу, оновлень та масштабування.

**Chroma** також є open-source рішенням під ліцензією Apache 2.0. Його можна запускати локально, self-hosted або використовувати як керований Chroma Cloud. Chroma надає збереження embeddings і metadata, dense-, sparse- та hybrid search, full-text search і metadata filtering. У реальному виборі я розглядав би Chroma насамперед для швидких локальних експериментів, невеликих RAG-пайплайнів або прототипування, де важливо максимально швидко перейти від ідеї до робочого тесту. Для задачі, де потрібен більший контроль над окремим векторним сервісом і його подальшим розгортанням, для мене більш природним вибором виглядав би Qdrant.

Таким чином, **Pinecone** є обґрунтованим вибором саме для цієї навчальної роботи: він мінімізує витрати часу на інфраструктуру та дозволяє перевірити повний пошуковий пайплайн у хмарному сервісі. Для власного проєкту з локальним зберіганням даних я, скоріше за все, обрав би **Qdrant**. **Chroma** розглядав би як зручний варіант для швидкого прототипування або локального експериментального RAG-сценарію.

##### 2. Вибір моделі `allenai/specter2_base`

У роботі використовується модель `allenai/specter2_base`, оскільки предметна область задачі — пошук саме за науковими публікаціями. На відміну від універсальної моделі `sentence-transformers/all-MiniLM-L6-v2`, модельне сімейство SPECTER2 створене для представлення наукових документів на основі їхніх назв та анотацій.

У картці моделі зазначено, що SPECTER2 здатна генерувати *“task specific embeddings for scientific tasks when paired with adapters”*. Також в описі наведено формати задач, на яких навчалося модельне сімейство: *“Proximity (Retrieval)”* та *“Adhoc Search”* серед інших. Це безпосередньо відповідає задачі цієї роботи: потрібно перетворити назви й анотації статей на embeddings, а потім знаходити релевантні наукові документи за текстовим запитом.

Модель `all-MiniLM-L6-v2` також може використовуватися для information retrieval і semantic similarity, але вона є універсальною sentence embedding model, навченою на великій суміші загальних текстових пар. Для пошуку за спеціалізованим корпусом наукових статей логічно обрати модельне сімейство, яке навчалося саме на наукових документах і citation-based даних.

Водночас є важливе технічне уточнення. В офіційній картці `allenai/specter2_base` вказано, що це базова модель, яку для найкращої якості на конкретній downstream-задачі рекомендовано використовувати разом із відповідним adapter. Зокрема, для retrieval/search-сценаріїв передбачені відповідні адаптери SPECTER2. У цій роботі використовується конфігурація `allenai/specter2_base`, визначена умовою завдання, щоб зберегти відтворюваність експерименту. Тому результати слід інтерпретувати як результати заданого навчального пайплайна, а не як максимально оптимізовану конфігурацію сімейства SPECTER2.

##### 3. Метрика схожості для embeddings

В офіційній картці моделі `allenai/specter2_base` я не знайшов явної універсальної рекомендації щодо вибору метрики для побудови векторного індексу. Проте у статті *SciRepEval: A Multi-Format Benchmark for Scientific Document Representations*, яка описує підхід, пов'язаний із модельним сімейством SPECTER2, наведено конкретний спосіб оцінювання embeddings для задачі **ad-hoc search**. У цій задачі текстовий запит використовується для ранжування релевантних наукових статей, що найближче відповідає сценарію даної роботи. Автори зазначають, що документи ранжуються за зростанням **Euclidean distance** між embedding запиту та embeddings кандидатів. Крім того, під час навчання для proximity- та ad-hoc-search задач у triplet loss також використовується Euclidean distance як міра відстані між векторами.

Водночас у цьому завданні embeddings нормалізуються під час генерації:

```python
normalize_embeddings=True
```

Тобто для кожного embedding виконується умова:

$$|x| = 1, \qquad |y| = 1.$$

Для таких векторів косинусна схожість дорівнює скалярному добутку:

$$\mathrm{cosine}(x, y) = \frac{x \cdot y}{|x| |y|} = x \cdot y.$$

Також квадрат евклідової відстані між нормалізованими векторами дорівнює:

$$\|x-y\|^2 = \|x\|^2 + \|y\|^2 - 2(x \cdot y) = 2 - 2\mathrm{cosine}(x, y).$$

Отже, за умови нормалізації і векторів документів, і вектора запиту, усі три варіанти повинні давати однакове ранжування результатів:

```text
більший cosine similarity
    ⇔ більший dot product
    ⇔ менша L2-distance
```

Таким чином, використання метрики `cosine` для основного індексу Pinecone не суперечить підходу зі статті, оскільки в нормалізованому просторі вона еквівалентна ранжуванню за Euclidean distance. Водночас вибір метрики є принципово важливим, якщо embeddings не нормалізовані: у такому випадку cosine оцінює переважно напрямок вектора, dot product залежить також від його довжини, а Euclidean distance визначає геометричну відстань у просторі та може формувати інше ранжування.

**Джерело:** Singh, A. et al. *SciRepEval: A Multi-Format Benchmark for Scientific Document Representations*. Для ad-hoc search автори ранжують документи за Euclidean distance між embedding запиту та embeddings кандидатів.

#### 1.3. Отримання ембеддингів

Для побудови векторних представлень наукових статей реалізовано скрипт `scripts/02_embed.py`. Він завантажує підготовлену вибірку з файлу `data/arxiv_subset.parquet` та формує для кожної статті текстовий вхід у форматі:

```python
title + " [SEP] " + abstract
```

Таким чином, модель отримує одночасно назву статті та її анотацію, розділені спеціальним маркером `[SEP]`. Саме назва та анотація містять основний зміст наукової публікації, який необхідний для подальшого семантичного пошуку.

Для генерації embeddings використовується модель:

```python
allenai/specter2_base
```

Тексти кодуються пакетно через метод `SentenceTransformer.encode()` із такими параметрами:

```python
embeddings = model.encode(
    texts,
    batch_size=64,
    show_progress_bar=True,
    normalize_embeddings=True,
)
```

Пакетна обробка дозволяє послідовно закодувати всі `10 000` текстів без необхідності одночасно завантажувати повний набір входів у пам'ять моделі. Параметр `normalize_embeddings=True` нормалізує кожен отриманий embedding до одиничної евклідової норми, що буде важливо під час подальшого порівняння метрик схожості.

Згенеровані вектори зберігаються у файлі:

```text
embeddings/embeddings.npy
```

Порядок векторів у масиві зберігається відповідно до порядку записів у `data/arxiv_subset.parquet`: embedding з індексом `i` відповідає статті у рядку `i` підготовленого DataFrame. Ідентифікатори статей та metadata не дублюються у файлі embeddings; вони будуть повторно завантажені з Parquet-файлу під час формування записів для Pinecone.

##### Технічне уточнення щодо конфігурації моделі

##### Чому для нормалізованих embeddings cosine similarity еквівалентна dot product?

Косинусна схожість двох векторів визначається як:

Косинусна схожість двох векторів визначається як:

$$\mathrm{cosine}(x, y) = \frac{x \cdot y}{\|x\| \|y\|}.$$

У цьому проєкті embeddings генеруються з параметром:

```python
normalize_embeddings=True
```

Тому кожен вектор після нормалізації має одиничну довжину:

$$\|x\| = 1, \qquad \|y\| = 1.$$

Після підстановки одиничних норм у формулу cosine similarity отримуємо:

Після підстановки одиничних норм у формулу cosine similarity отримуємо:

$$\mathrm{cosine}(x, y) = \frac{x \cdot y}{1 \cdot 1} = x \cdot y.$$

Отже, для нормалізованих embeddings значення `cosine similarity` і `dot product` є однаковими. Відповідно, якщо один і той самий нормалізований embedding запиту порівнюється з однаковим набором нормалізованих embeddings документів, обидві метрики повинні сформувати однакове ранжування результатів пошуку, за винятком можливих незначних числових відхилень, пов'язаних з обчисленнями з плаваючою точкою.

Цей факт є важливим для подальшого експерименту: якщо результати пошуку за `cosine similarity` і `dot product` суттєво відрізнятимуться, це може вказувати на відсутність нормалізації векторів або на помилку в реалізації порівняння.


### 2. Побудова embeddings

Для збереження embeddings і подальшого виконання семантичного пошуку реалізовано скрипт `scripts/03_load_to_pinecone.py`.

Скрипт виконує такі дії:

1. Завантажує підготовлені metadata статей із файлу `data/arxiv_subset.parquet`.
2. Завантажує попередньо сформовані embeddings із файлу `embeddings/embeddings.npy`.
3. Перевіряє відповідність кількості статей кількості embeddings та очікувану розмірність векторів.
4. Створює serverless-індекс Pinecone, якщо він ще не існує.
5. Формує записи з embedding та metadata для кожної статті.
6. Завантажує вектори до Pinecone пакетами по `200` записів.
7. Перевіряє загальну кількість збережених векторів у створеному індексі.

#### 3.4. Відповіді на теоретичні запитання

##### Чи збігаються top-5 для cosine similarity і dot product і чому?

Так, у проведеному експерименті top-5 документів для `cosine similarity` і `dot product` повністю збігаються. Більше того, збігаються і числові значення scores.

Причина полягає в тому, що embeddings документів та embedding запиту були нормалізовані за допомогою параметра:

```python
normalize_embeddings=True
```

Для нормалізованих векторів виконується:

$$|x| = 1, \qquad |y| = 1.$$

Косинусна схожість визначається формулою:

$$\mathrm{cosine}(x, y) = \frac{x \cdot y}{|x| |y|}.$$

Оскільки обидві норми дорівнюють одиниці:

$$\mathrm{cosine}(x, y) = x \cdot y.$$

Отже, у нормалізованому просторі `cosine similarity` і `dot product` є еквівалентними та повинні формувати однакове ранжування результатів.

##### Чи відрізняються результати для L2-distance і чому?

У проведеному експерименті результати для `L2-distance` не відрізняються за порядком документів: top-5 і порядок ранжування повністю збігаються з результатами `cosine similarity` та `dot product`.

Відрізняються лише значення самої метрики та напрямок їх інтерпретації:

```text
cosine similarity / dot product → більше значення є кращим;
L2-distance                    → менше значення є кращим.
```

Для нормалізованих векторів квадрат евклідової відстані пов'язаний із cosine similarity формулою:

$$|x-y|^2 = |x|^2 + |y|^2 - 2(x \cdot y) = 2 - 2\mathrm{cosine}(x, y).$$

Це монотонний зв'язок: чим більша cosine similarity, тим меншою є L2-distance. Тому нормалізовані embeddings дають однакове ранжування для всіх трьох метрик.

##### Що сталося б, якби embeddings не були нормалізовані?

Без нормалізації метрики вже не були б еквівалентними та могли б сформувати різні top-5 результати.

* `Cosine similarity` оцінювала б переважно кут між векторами, тобто схожість їх напрямків, майже не враховуючи абсолютну довжину.
* `Dot product` залежав би як від напрямку, так і від норм векторів. Документи з embeddings більшої довжини могли б отримувати підвищений score навіть за неідеальної семантичної відповідності.
* `L2-distance` оцінювала б абсолютну геометричну відстань між векторами, яка також залежить від їхніх довжин.

Отже, без нормалізації вибір метрики міг би суттєво змінити результати пошуку. Нормалізація в цьому проєкті робить порівняння метрик прозорим і дозволяє експериментально підтвердити їхню еквівалентність для одиничних embeddings.

### 4. Chunking довгих анотацій і пошук за фрагментами

Для дослідження впливу chunking на якість пошуку реалізовано скрипт `scripts/05_chunking.py`. У межах експерименту було відібрано `30` статей із найдовшими анотаціями за кількістю слів. Для кожної статті побудовано два варіанти фрагментації тексту:

1. **Fixed-size chunking** — механічне розбиття анотації на чанки по `100` слів із перекриттям `20` слів.
2. **Sentence-based chunking** — об'єднання повних речень у чанки розміром до `100` слів без розриву речень.

У термінології завдання друга стратегія розглядається як semantic chunking, однак у реалізації вона є саме sentence-aware підходом: межі чанків визначаються межами речень, а не окремим аналізом семантичної близькості між частинами тексту.

#### 4.1. Параметри експерименту

| Параметр                                 |                Значення |
| ---------------------------------------- | ----------------------: |
| Кількість вибраних статей                |                    `30` |
| Розмір fixed-size chunk                  |              `100` слів |
| Overlap для fixed-size chunking          |               `20` слів |
| Максимальний розмір sentence-based chunk |              `100` слів |
| Модель embeddings                        | `allenai/specter2_base` |
| Розмірність embeddings                   |                   `768` |
| Метрика Pinecone                         |                `cosine` |
| Індекс fixed-size chunks                 |    `arxiv-chunks-fixed` |
| Індекс sentence-based chunks             | `arxiv-chunks-semantic` |

Для кожного чанка embedding формувався на основі тексту:

```python
embedding_text = title + " [SEP] " + chunk_text
```

Додавання назви статті до кожного фрагмента дозволяє зберегти загальний тематичний контекст документа під час пошуку за окремими частинами анотації.

#### 4.2. Порівняння стратегій розбиття

За результатами обробки `30` найдовших анотацій отримано такі характеристики:

| Стратегія               | Кількість chunks | Мінімальний розмір, слів | Максимальний розмір, слів | Середній розмір, слів |
| ----------------------- | ---------------: | -----------------------: | ------------------------: | --------------------: |
| Fixed-size chunking     |            `121` |                     `18` |                     `100` |                `91.3` |
| Sentence-based chunking |            `121` |                     `16` |                     `100` |                `76.3` |

Обидві стратегії випадково сформували однакову кількість векторів — по `121`. Водночас структура цих чанків суттєво відрізняється.

**Fixed-size chunking** формує більш рівномірні за розміром фрагменти. Середній розмір чанка становить `91.3` слова, оскільки більшість фрагментів близькі до заданого ліміту у `100` слів. Проте такий підхід не враховує структуру речень та може розривати одну думку між сусідніми фрагментами. Крім того, overlap у `20` слів призводить до дублювання тексту в сусідніх chunks.

Наприклад, межа між двома fixed-size chunks розриває речення:

```text
End of chunk 1:
...The details of SN 1987A strongly suggest that it resulted from a merger of

Start of chunk 2:
of 5e40 ergs/s at day 20. The details of SN 1987A strongly suggest that it
resulted from a merger of 2 stellar cores...
```

У цьому прикладі початок другого чанка повторює частину попереднього фрагмента через overlap, а речення про походження `SN 1987A` розділяється механічною межею.

**Sentence-based chunking** зберігає повні речення. Через це середній розмір чанка нижчий — `76.3` слова: якщо наступне речення не вміщується в заданий ліміт, воно переноситься до нового фрагмента повністю. Такий підхід менш ефективно заповнює максимальний розмір чанка, але краще зберігає локальну читабельність і завершеність висловлювань.

Для тієї самої статті межа sentence-based chunks проходить між повними реченнями:

```text
End of chunk 1:
...an upper limit for the optical luminosity of the MS of 5e40 ergs/s at day 20.

Start of chunk 2:
The details of SN 1987A strongly suggest that it resulted from a merger of
2 stellar cores...
```

Для кожної стратегії створено окремий Pinecone-індекс:

| Індекс                  | Стратегія               | Кількість векторів |
| ----------------------- | ----------------------- | -----------------: |
| `arxiv-chunks-fixed`    | Fixed-size chunking     |              `121` |
| `arxiv-chunks-semantic` | Sentence-based chunking |              `121` |

Результат виконання завантаження:

```text

Total vectors in 'arxiv-chunks-semantic': 121
```

Таким чином, для обох стратегій було сформовано окремі векторні представлення тих самих `30` статей, що дозволяє порівнювати вплив саме способу розбиття тексту.

#### 4.6. Відповіді на теоретичні запитання

##### Яка стратегія дає більш осмислені чанки?

У межах цієї роботи під стратегією `semantic chunking` реалізовано sentence-based підхід: текст розбивається на речення, після чого повні речення об'єднуються у чанки до досягнення заданого ліміту слів. Це не є повноцінним семантичним розбиттям на основі embeddings або обчислення зміни теми між фрагментами тексту.

Для поточного експерименту більш осмислені та читабельні чанки формує саме sentence-based стратегія, оскільки вона зберігає завершені речення і не розриває думку на довільній межі за кількістю слів. Fixed-size chunking, навпаки, дає більш передбачуваний і рівномірний розмір фрагментів, але може розривати речення та частково дублювати текст через overlap.

Отримані результати пошуку також показали перевагу sentence-based підходу для вибраного запиту: у top-5 з'явилася третя релевантна стаття про спостереження `GRB 060614`, тоді як fixed-size видача містила фрагменти лише двох статей. Водночас цей висновок стосується конкретного невеликого експерименту і не може автоматично узагальнюватися на всі корпуси та запити.

##### Чи є випадки розрізаних речень і як це впливає на embeddings?

Так, у результатах fixed-size chunking було зафіксовано випадок, коли межа чанка проходить усередині речення. Наприклад, один фрагмент завершується словами:

```text
...The details of SN 1987A strongly suggest that it resulted from a merger of
```

а наступний chunk містить продовження цієї самої думки:

```text
...The details of SN 1987A strongly suggest that it resulted from a merger of
2 stellar cores...
```

У sentence-based стратегії для того самого тексту перший chunk завершується повним реченням, а наступний починається з нового речення.

Це впливає на embeddings, оскільки embedding формується на основі конкретного тексту чанка. Якщо речення розрізане, один із chunks може містити незавершену думку та гірше представляти її семантичний зміст. У fixed-size підході overlap частково компенсує цю проблему, оскільки частина тексту повторюється у сусідньому чанку, але водночас це збільшує дублювання інформації та може призводити до появи кількох дуже близьких фрагментів однієї статті у результатах пошуку.

##### Як розмір overlap впливає на кількість чанків і покриття тексту?

У fixed-size chunking наступний chunk починається через крок:

```python
step = chunk_size - overlap
```

У проведеному експерименті використано:

```python
CHUNK_SIZE = 100
OVERLAP = 20
```

тому кожний наступний chunk починається через `80` слів після попереднього. Чим більшим є overlap, тим меншим стає крок між сусідніми чанками та тим більша частина контексту дублюється між ними.

Збільшення overlap має перевагу: інформація поблизу межі чанків рідше втрачається, а розірване речення або важливий термін можуть бути присутні у сусідньому фрагменті повніше. Проте за це доводиться платити збільшенням кількості або обсягу chunks, кількості embeddings, розміру індексу та часу обробки. Крім того, надто великий overlap може зменшувати різноманітність пошукової видачі, оскільки кілька дуже схожих chunks одного документа можуть займати верхні позиції ranking.

Таким чином, overlap є компромісом між збереженням контексту на межах фрагментів і надлишковим дублюванням даних. Його оптимальний розмір залежить від довжини документів, типу запитів і вимог до якості та вартості пошуку.


### 5. Гібридний пошук: BM25, vector search та Reciprocal Rank Fusion

Для порівняння лексичного, семантичного та гібридного пошуку реалізовано скрипт `scripts/06_hybrid_search.py`.

У цьому етапі використовуються три способи отримання результатів:

1. **BM25 search** — локальний лексичний пошук за токенами у полях `title` та `abstract`.
2. **Vector search** — семантичний пошук через основний Pinecone-індекс `arxiv-papers`.
3. **Hybrid search** — об'єднання двох ranked lists за допомогою `Reciprocal Rank Fusion (RRF)`.

#### 5.1. Реалізація BM25

Для BM25 формується локальний текстовий корпус із назв та анотацій усіх `10 000` статей:

```python
corpus = (
    data["title"].map(normalize_whitespace)
    + " "
    + data["abstract"].map(normalize_whitespace)
).tolist()
```

Токенізація виконується через регулярний вираз:

```python
def tokenize(text: str) -> list[str]:
    return re.findall(r"[a-z0-9]+(?:-[a-z0-9]+)*", text.lower())
```

Такий варіант приводить текст до нижнього регістру, прибирає пунктуацію та зберігає складені токени на кшталт `fine-tuning`.

BM25 повертає документи з найбільшим lexical score. На відміну від vector search, цей метод орієнтується на буквальні збіги термінів і не враховує, що одне й те саме слово може мати різне предметне значення.

#### 5.2. Реалізація гібридного пошуку через RRF

Для кожного запиту спочатку отримуються по `10` кандидатів із BM25 та Pinecone:

```python
RETRIEVAL_TOP_K = 10
```

Після цього результати об'єднуються через Reciprocal Rank Fusion:

```text
RRF_score(document) = Σ 1 / (k + rank)
```

У базовому експерименті використано:

```python
DEFAULT_RRF_K = 60
```

RRF використовує не абсолютні scores методів, а лише позиції документів у кожному ranked list. Це важливо, оскільки значення BM25 score та vector similarity score мають різну природу й не можуть коректно додаватися безпосередньо.

Документ, який присутній у результатах обох методів, отримує два внески до RRF score і може піднятися вище за документи, знайдені лише одним пошуковим механізмом.

#### 5.8. Відповіді на теоретичні запитання

##### Який метод дав кращий результат і чому?

За результатами виконаних експериментів не можна стверджувати, що один із методів стабільно забезпечив якісно релевантну видачу для всіх трьох обов'язкових запитів. Основною причиною є обмеження сформованого корпусу: поточний dataset subset містить лише статті за `2007` рік, тоді як запити `BERT fine-tuning` та `Yann LeCun convolutional networks` орієнтовані на іншу або значно пізнішу предметну область.

Для запиту `BERT fine-tuning` BM25 знайшов точний лексичний збіг `fine-tuning`, але у фізичному, а не ML-контексті. Vector search також не повернув тематично релевантних документів, а hybrid search не зміг покращити результат, оскільки якісних кандидатів не було в обох вхідних списках.

Для запиту `Yann LeCun convolutional networks` hybrid search підняв на перше місце документ `Optimization in Gradient Networks`, оскільки він був присутній у результатах обох методів (`BM25 rank = 4`, `Vector rank = 6`). Однак цей документ не стосується convolutional neural networks або Yann LeCun. У цьому випадку RRF продемонстрував механіку об'єднання результатів, але фактично підсилив спільну помилку двох методів.

Найбільш переконливий відносний результат hybrid search було отримано для запиту `making computers understand human emotions from text`. Документ `On the Development of Text Input Method - Lessons Learned` був знайдений обома методами (`BM25 rank = 3`, `Vector rank = 2`) і піднявся на першу позицію після RRF. Він належить до категорії `cs.CL` та пов'язаний з обробкою тексту, тому виглядає ближчим до змісту запиту, ніж більшість інших результатів. Водночас він також не є повністю релевантним, оскільки присвячений методам введення тексту, а не визначенню емоцій у текстових даних.

Отже, у межах проведеного експерименту hybrid search дав найкращий відносний результат для третього запиту, оскільки підняв документ, підтриманий одночасно лексичним і семантичним пошуком. Проте загальну перевагу hybrid search за якістю встановити неможливо через відсутність достатньо релевантних документів у поточному корпусі.

##### Чи є документи в top-5 гібридного пошуку, яких немає в top-5 окремих методів, і чому?

У фактичних результатах проведеного експерименту документів, які потрапили до top-5 hybrid search і при цьому були відсутні одночасно в top-5 BM25 та top-5 vector search, не виявлено.

Для запиту `making computers understand human emotions from text` перший результат hybrid search — `On the Development of Text Input Method - Lessons Learned` — уже був присутній у top-5 обох методів:

```text
BM25 rank = 3
Vector rank = 2
```

Для запиту `Yann LeCun convolutional networks` перший hybrid-результат `Optimization in Gradient Networks` був присутній у top-5 BM25 (`rank = 4`), але у vector search займав шосту позицію й не відображався серед видимих top-5 результатів.

Водночас така ситуація теоретично можлива, оскільки для RRF у реалізації беруться не лише перші п'ять, а по `10` кандидатів від кожного методу:

```python
RETRIEVAL_TOP_K = 10
```

Наприклад, документ, що займає шосте місце в BM25 і шосте місце у vector search, може отримати два внески до RRF score і після об'єднання потрапити до фінального top-5, випередивши документи, які високо ранжуються лише одним методом. Таким чином, RRF може вивести у фінальну top-5 видачу документ, який не був видимим у скороченому top-5 жодного окремого пошуку, якщо цей документ підтримується обома ranked lists.

##### Як зміна параметра `k` у RRF впливає на видачу?

У Reciprocal Rank Fusion підсумкова оцінка документа обчислюється за формулою:

$$
\mathrm{RRF}(d) = \sum_{r \in R(d)} \frac{1}{k + r},
$$

де $r$ — позиція документа в одному з ranked lists, а $k$ — параметр згладжування.

Чим меншим є значення `k`, тим сильніше RRF розрізняє верхні позиції рейтингу. Наприклад, при `k = 1` документ на першій позиції одного методу отримує значно більший внесок порівняно з документом на п'ятій або десятій позиції. Чим більшим є значення `k`, тим більше згладжується різниця між окремими рангами, а присутність документа у кількох ranked lists стає відносно важливішою.

Для запиту `making computers understand human emotions from text` було порівняно значення `k = 1`, `k = 10`, `k = 60` та `k = 100`.

| Rank | Title                                                       |  `k = 1` | `k = 10` | `k = 60` | `k = 100` |
| ---: | ----------------------------------------------------------- | -------: | -------: | -------: | --------: |
|    1 | On the Development of Text Input Method - Lessons Learned   | 0.583333 | 0.160256 | 0.032002 |  0.019513 |
|    2 | An Automated Evaluation Metric for Chinese Text Entry       | 0.500000 | 0.090909 | 0.016393 |  0.009901 |
|    3 | Opinion Dynamics and Sociophysics                           | 0.500000 | 0.090909 | 0.016393 |  0.009901 |
|    4 | Towards Understanding the Origin of Genetic Languages       | 0.333333 | 0.083333 | 0.016129 |  0.009804 |
|    5 | Extracting the hierarchical organization of complex systems | 0.250000 | 0.076923 | 0.015873 |  0.009709 |

У проведеному експерименті зміна параметра `k` не змінила порядок top-5 документів навіть для дуже малого значення `k = 1`. В усіх варіантах перше місце посідає документ `On the Development of Text Input Method - Lessons Learned`, оскільки він присутній у результатах обох методів на високих позиціях:

```text
BM25 rank = 3
Vector rank = 2
```


При `k = 1` документ, знайдений обома методами, отримує:

$$
\frac{1}{1 + 3} + \frac{1}{1 + 2}
=
\frac{1}{4} + \frac{1}{3}
=
0.583333.
$$

тоді як документ, що займає перше місце лише в одному зі списків, отримує:

$$
\frac{1}{1 + 1}
=
0.500000.
$$


Тому навіть при сильному пріоритеті верхніх окремих позицій сумарна підтримка двох retrieval-методів залишається достатньою, щоб спільний документ очолював hybrid ranking.

Отже, параметр `k` впливає на величину RRF score та на баланс між перевагою високого одиничного рангу і підтримкою документа кількома методами. У поточному експерименті структура результатів виявилася стабільною: зміна `k` змінює числові оцінки, але не змінює фінальний порядок документів.


## 6. Аналіз і висновки

### 6.1. Семантичний пошук vs BM25

BM25 і семантичний vector search вирішують різні типи задач. BM25 сильний у випадках, коли релевантність визначається точним збігом термінів: назвами методів, іменами авторів, абревіатурами або специфічними ключовими словами. Vector search, навпаки, має перевагу для перефразованих запитів, де зміст документа може відповідати потребі користувача без буквального повторення слів із запиту.

У проведеному експерименті запит `BERT fine-tuning` показав слабкість BM25 при неоднозначності термінів. Метод знайшов статті з точним словосполученням `fine-tuning`, але воно використовувалося у фізичному контексті, а не в контексті навчання нейронних мереж. Це не є помилкою BM25: метод коректно виконав лексичний пошук, але не може самостійно розрізнити предметний зміст однакових термінів.

Для запиту `making computers understand human emotions from text` vector search повернув документи, пов'язані з opinion dynamics, text technologies та collective attention, тобто частково наблизився до загального змісту запиту без точного збігу всіх слів. Hybrid search через RRF підняв документ `On the Development of Text Input Method - Lessons Learned`, який був знайдений і BM25, і vector search. Хоча цей документ не є повністю релевантним задачі emotion analysis, саме цей приклад найкраще демонструє ідею гібридного пошуку: лексична та семантична підтримка одного документа можуть підсилювати одна одну.

Загальне практичне правило можна сформулювати так:

* **BM25** варто використовувати як основний або обов'язковий компонент для запитів із точними термінами, назвами, кодами, іменами, абревіатурами та цитатами.
* **Vector search** корисніший для запитів природною мовою, перефразувань, синонімів і пошуку за змістом.
* **Hybrid search** доцільний у прикладних системах, де одночасно важливі і точні збіги, і семантична близькість, але його якість усе одно залежить від наявності релевантних документів у корпусі.

У цій роботі неможливо чесно зробити висновок про абсолютну перевагу одного методу, оскільки підготовлений subset містить лише статті `2007` року, а частина заданих запитів орієнтована на пізніші AI/NLP-теми.

### 6.2. Вплив розміру чанка

Розмір чанка визначає, скільки контексту міститиме один embedding і наскільки точно пошук зможе локалізувати релевантний фрагмент документа.

Якщо chunk занадто малий, наприклад `10–15` слів, він може втратити контекст. Окремий термін або короткий фрагмент речення часто не містить достатньої інформації для формування якісного embedding. У такому випадку пошук може знаходити збіги за окремими словами, але гірше розуміти, про що саме йдеться у фрагменті. Також значно зростає кількість chunks, embeddings та записів у векторному індексі.

Якщо chunk занадто великий, наприклад `500+` слів, один embedding починає представляти одразу кілька підтем або аргументів документа. Релевантний короткий фрагмент може «розчинитися» в загальному змісті великого чанка. Крім того, необхідно враховувати максимальну довжину входу embedding-моделі: занадто довгий текст може бути обрізаний токенізатором, і частина змісту взагалі не вплине на embedding.

У цьому експерименті використано цільовий розмір `100` слів. Fixed-size chunking із overlap `20` слів дав середній розмір `91.3` слова, але допускав розрив речень і дублювання контексту. Sentence-based chunking дав менший середній розмір — `76.3` слова, проте зберігав цілі речення та продемонстрував більш читабельну і дещо різноманітнішу видачу для тестового запиту.

Універсального оптимального розміру чанка не існує. Він залежить від типу документів, типових запитів, моделі embeddings та того, чи потрібен пошук за окремими фактами, абзацами або загальним змістом документа. Для реальної системи розмір і стратегію chunking потрібно підбирати експериментально на наборі репрезентативних запитів із розміченими релевантними результатами.

### 6.3. Невідповідна метрика: Euclidean distance для нормалізованих embeddings

У межах цієї роботи embeddings документів і запитів нормалізуються до одиничної довжини:

```python
normalize_embeddings=True
```

Тому використання Pinecone-індексу з метрикою `euclidean` замість `cosine` не змінило б порядок результатів за умови, що і документи, і query embeddings залишаються нормалізованими.

Для двох векторів одиничної довжини $x$ і $y$:

$$
\|x\| = 1, \qquad \|y\| = 1.
$$

Квадрат евклідової відстані між ними дорівнює:

$$
\|x-y\|^2
=
(x-y) \cdot (x-y)
=
\|x\|^2 + \|y\|^2 - 2(x \cdot y).
$$

Оскільки обидві норми дорівнюють одиниці:

$$
\|x-y\|^2
=
2 - 2(x \cdot y).
$$

Для нормалізованих векторів скалярний добуток дорівнює cosine similarity:

$$
x \cdot y
=
\mathrm{cosine}(x,y).
$$

Отже:

$$
\|x-y\|^2
=
2 - 2\mathrm{cosine}(x,y).
$$

Цей зв'язок є монотонним: чим більшою є cosine similarity, тим меншою є Euclidean distance.

```text
ranking by maximum cosine similarity
=
ranking by maximum dot product
=
ranking by minimum Euclidean distance
```

Саме це було підтверджено локальним exact search у частині 3: `cosine`, `dot product` і `L2-distance` повернули однаковий top-5 документів.

Якби embeddings не були нормалізовані, така еквівалентність зникла б. У такому випадку `dot product` та `L2-distance` залежали б не лише від напрямку векторів, але й від їхньої довжини, тому вибір метрики міг би суттєво змінити ranking.

### 6.4. Обмеження Pinecone Starter і масштабування рішення

У межах виконання роботи було практично виявлено одне обмеження безкоштовного плану Pinecone: при спробі створити serverless-індекс у регіоні AWS `eu-west-1`, який географічно ближчий для користувача в Європі, сервіс повернув помилку про недоступність цього регіону для free plan. Тому для виконання завдання індекси були створені в доступному регіоні `us-east-1`.

Для навчального експерименту з `10 000` статей це обмеження не є критичним. Проте в production-сценарії вибір регіону впливає на latency, вимоги щодо місця обробки даних і потенційні політики зберігання інформації. Якщо система працює з європейськими користувачами або даними, регіон розгортання повинен обиратися відповідно до нефункціональних вимог, а не лише за доступністю безкоштовного тарифу.

Якби корпус містив не `10 000`, а `10` мільйонів статей, підхід також потребував би перегляду. Насамперед не слід було б генерувати всі embeddings та завантажувати їх одним локальним процесом без контролю ресурсів. Потрібен був би batch-oriented ingestion pipeline із можливістю відновлення після помилок, журналюванням оброблених документів, повторним запуском окремих batch-ів та контролем версії embedding-моделі.

Для такого масштабу необхідно було б окремо оцінити:

* вартість зберігання десятків мільйонів embeddings та metadata;
* вартість масової генерації embeddings;
* latency і throughput пошуку;
* потребу у metadata-фільтрах, namespaces або розподілі індексів;
* стратегію оновлення або переіндексації документів;
* вимоги щодо регіону зберігання даних і контролю інфраструктури.

Залежно від вимог до безпеки, вартості й контролю над інфраструктурою, для великого production-рішення я б розглядав не лише керований Pinecone, але й self-hosted або cloud-managed варіанти на основі Qdrant чи Milvus. Якщо ж у системі вже є PostgreSQL і векторний пошук не є домінуючим високонавантаженим сценарієм, практичним варіантом може бути `pgvector`, щоб не вводити окрему векторну базу даних без достатньої потреби.

Таким чином, Pinecone Starter добре підходить для швидкого навчального прототипу, але результати цього завдання не можна безпосередньо переносити на production-систему з мільйонами документів без окремої оцінки масштабованості, вартості, latency, регіональних і безпекових обмежень.
